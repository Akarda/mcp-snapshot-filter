How AI Models Interpret Accessibility Snapshots

When an AI model receives a page snapshot, it doesn't see pixels — it sees the accessibility tree: a structured text representation of every semantic element on the page. From this text, the model reconstructs a mental model of the page layout and content.

From text to understanding

The model uses roles as a blueprint — navigation, main, heading, table, button map directly to well-known UI patterns. A navigation with link children means a nav bar. A table with columnheader and row children means a data grid.

Hierarchy implies layout — indentation and parent-child nesting tell the model spatial relationships. Children of main are the page content area. Children of navigation are nav items.

Names and values provide content — accessible names like "Dashboard", "Submit", "Status" tell the model what the user sees as text on each element.

Common patterns are recognized — a banner at the top + main in the middle + contentinfo at the bottom is a standard page layout. A searchbox near a table implies a filter/search UI.

What the AI can understand

  • Page structure and element hierarchy
  • Semantic meaning of each element (role)
  • Text content and labels
  • Interactive elements (buttons, inputs, links)
  • Data structures (tables, lists)
  • Navigation and landmark regions
  • Form fields and their purposes

What the AI cannot see

  • Visual styling (colors, fonts, spacing)
  • Exact positioning (is the nav horizontal or vertical?)
  • Images (only alt text descriptions)
  • Custom components without semantic roles
  • Relative proportions (sidebar width, etc.)
  • CSS animations or visual state changes
  • Decorative or presentational elements

Why filtering matters

Real-world pages produce large accessibility trees with many redundant nodes — deeply nested generic containers, inline text fragments, decorative elements. Filtering removes this noise while preserving the semantic cues the AI relies on: roles, hierarchy, names, and values.
